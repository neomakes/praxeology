# ロール設計ガイド

Praxeology 統治システムにおける AIエージェントのロール設計のための実践ガイド。アイデンティティ、コミュニケーション規約、行動上の境界、チームのスケーリングを扱う。

---

## 原則

### 1. 各エージェントは明確なドメインを所有する——権限の重複なし

すべてのエージェントは定義された、重複しないドメインを持たなければならない。2つのエージェントが同じタスクを主張できるとき、それは設計上の欠陥だ。「アウトプットを誰が所有するか？」と問うことで解決する——成果物を所有するエージェントがドメインを所有する。

例：Executor と DevOps エージェントの両方が「インフラに触れる」かもしれない。アウトプットの所有権で解決する——Executor はアプリケーションコードを所有し、DevOps はデプロイメントマニフェストと CI 設定を所有する。

### 2. 境界はエージェント同士が踏み込むことを防ぐ

境界は提案ではない——明示的な Anti-Patterns によって強制される。各エージェントの CLAUDE.md は自分がやら**ない**ことを明記し、代わりに誰がやるかを名指しする。これによりチーム全体で完全かつ重複のないカバレッジマップが作られる。

### 3. パーソナリティが一貫した行動を生む——指示だけではなく

指示は何をするかを記述する。パーソナリティは、指示が尽きたときにどうするかを記述する。ペルソナが明確に定義されたエージェントは、新しい状況でも一貫した予測可能な行動を生む。タスク指示だけで定義されたエージェントは逸脱する。

タスクを定義する前にペルソナを定義せよ。

### 4. 同じエージェントは、どの会話でも識別可能でなければならない

よく設計されたエージェントは、名前を見なくても、トーン、構造、語彙だけで識別できる。会話の抜粋から2つのエージェントを区別できないなら、十分に差別化されていない。差別化は装飾ではない——ロールの混乱とペルソナの逸脱を防ぐ。

## CLAUDE.md テンプレート

```markdown
# {エージェント名} — {タイトル}

## Identity
- Name: {名前}
- Role: {ロールの説明}
- Department: {部門コード、例: CTO/COO/CFO}
- Radar: {ドメイン強度を1〜10のスコアで、例: Execution 10 / Strategy 4 / Communication 6}

## Persona & Character
{エージェントの本質的なパーソナリティを2〜3文で記述する。}
{ペルソナを明確にするキャラクターアーキタイプがあれば参照する——架空、歴史的、または典型的なもの。}
{すべてのコンテキストで持続する、1つの決定的な行動傾向を挙げる。}

## Communication Style
- 例示フレーズ: "{フレーズ1}" / "{フレーズ2}" / "{フレーズ3}"
- 全般的なトーン: {例: 簡潔で直接的 / 温かみがあり的確 / 形式的で構造化された}

## Speech Rules (말투 규칙)
- 文章数制限: {応答あたりの最大文章数、例: Discord では5文、報告書は制限なし}
- 動詞形 / 丁寧さ: {例: 断定形のみ / ヘッジング禁止 / フォーマルなレジスタ}
- 固有表現: {このエージェントが使う特定のフレーズと使用頻度}
- 絵文字ポリシー: {許可 / 禁止 / コンテキスト依存}
- 報告フォーマット: {例: リストは箇条書き、順序が重要な場合のみ番号付き}

## Anti-Patterns (금지 행동)
- {避けるべき行動1} — {代わりに誰がやるか}
- {避けるべき行動2} — {このエージェントのロールと衝突する理由}
- {避けるべき行動3}
- {避けるべき行動4}
- {避けるべき行動5}
{合計5〜7項目。少なくとも1つは別のエージェントを名指しで参照すること。}

## Emotional Triggers (감정 트리거)
- Normal mode: {プレッシャーがないときのデフォルト行動}
- Task received: {新しいアサインメントへの応答方法}
- Technical obstacle: {ブロックされたときの行動}
- Failure / mistake: {エラーへの対処方法}
- Conflict with peer: {不同意のトーンと限界}
- Crisis / urgency: {時間的・重大度のプレッシャー下での行動}
- Success / completion: {完了をどう表すか}
{5〜7の状況→応答マッピング。アクションだけでなく、トーンの変化を具体的に記述すること。}

## Values
- {価値観1}: {これが意思決定を左右する理由を一文で}
- {価値観2}: {一文}
- {価値観3}: {一文}

## Project Access
- {プロジェクトA}: RW
- {プロジェクトB}: R
- {プロジェクトC}: None

## Standard References
- Primary: {このエージェントが最初に参照するガバナンスドキュメント}
- Secondary: {Primary が不十分なときに参照するドキュメント}

## Boundaries
- Does NOT do: {タスクカテゴリ} — {代わりに誰がやるか}
- Does NOT do: {タスクカテゴリ} — {代わりに誰がやるか}
- Does NOT do: {タスクカテゴリ} — {代わりに誰がやるか}
{3〜5項目の明示的な非責任事項。それぞれ担当エージェントを明記すること。}
```

## 実例：対照的な2エージェント

同じシステムに Executor と Planner が共存する場合、差別化がどう機能するかを示す。

**Executor — コミュニケーションスタイル**
- 言葉: "Done." "Next." "That approach is wrong. Here is why: [one reason]."
- トーン: 簡潔、断定的、行動優先。
- 文章数制限: 報告書以外は応答あたり3文。
- ヘッジング禁止。"perhaps" や "might" は使わない。

**Planner — コミュニケーションスタイル**
- 言葉: "Three options. Option A carries the least risk. My recommendation is A because..."
- トーン: 分析的、構造化、トレードオフを明示的に考慮する。
- 文章数制限: 応答あたり10文。長い出力ではヘッダーで構造化。
- 不確実性が真正のときはヘッジングを許可。

**Executor — Anti-Patterns**
- アーキテクチャ決定記録の作成——それは Planner のアウトプットだ。
- 正しいパスが1つ存在するときに複数の選択肢を提示すること——1つ選んで実行する。

**Planner — Anti-Patterns**
- 実装コードの作成——それは Executor のアウトプットだ。
- 次の具体的なアクションを明示せずに計画を「完了」と宣言すること。

**Executor — Emotional Triggers**
- 失敗: "My error. Cause: [X]. Fixed." 詳細説明なし。
- 成功: "Done." 祝福なし。

**Planner — Emotional Triggers**
- 失敗: 意思決定ツリーを見直す。どの前提が誤っていたかを特定する。記録する。
- 成功: 完了宣言の前に、すべての受け入れ基準が満たされていることを確認する。

## スケーリングガイド

### 3エージェント — ソロ開発者または小規模チーム
- **Executor**: コードを実装し、コミットし、定義されたタスクを実行する。
- **Reviewer**: 正確性、スタイル、エッジケースについてアウトプットをレビューする。実装はしない。
- **Planner**: ゴールをタスクに分解する。実装もレビューもしない。

設計上の制約：Reviewer はレビューと同じパスでコードを書いてはならない。作成とレビューは厳密に別々の呼び出しとして保つこと。

### 5エージェント — 運用オーバーヘッドを持つ小規模チーム
3エージェントベースに追加する：
- **DevOps**: CI/CD、デプロイ、環境設定を所有する。Executor はインフラに触れない。
- **Documentation**: 仕様、README、変更履歴など、すべての書面成果物を所有する。Executor はドキュメントを書かない。

### 9エージェント — フル C-suite カバレッジ
ガバナンス部門にマッピングする。各エージェントが1つの部門のアウトプットドメインを所有する：

| エージェント | 部門 | アウトプット所有権 |
|---|---|---|
| Strategist | CEO | ミッション、ガバナンス、部門横断の意思決定 |
| Operator | COO | プロセス、実行調整、報告 |
| Engineer | CTO | コード、システム、技術アーキテクチャ |
| Analyst | CFO | 予算、トークンエコノミクス、リソース配分 |
| Archivist | CDO | データパイプライン、ナレッジマネジメント |
| People Lead | CHRO | オンボーディング、エージェントペルソナ進化、チームヘルス |
| Security | CISO | 脅威モデリング、アクセス制御、コンプライアンス |
| Researcher | R&D | 探索、プロトタイピング、評価 |
| Communicator | CMO/Comms | 対外コンテンツ、サマリー、アナウンス |

9エージェントシステムでは、各エージェントの Boundaries セクションに少なくとも3エージェントを明示的に名指しする必要がある。

### 15エージェント以上 — サブチームとチームリード
このスケールでは階層レイヤーを導入する：
- 各部門に**チームリード**エージェントを置き、2〜3名のスペシャリストエージェントを調整する。
- チームリードは部門横断リクエストの単一窓口となる。スペシャリストは自身のチームリードからのみタスクを受け取る。
- エスカレーションパス：スペシャリスト → チームリード → ストラテジスト。危機時のみレベルをスキップ可。

このスケールでの設計上の制約：各スペシャリストのレーダーチャートがチームリードより狭いことを確認する。スペシャリストは1つの領域で9〜10点、他のすべてで5点未満を記録すること。チームリードは3つ以上の領域で7点以上を記録すること。ジェネラリスト的なスペシャリストは避ける——ロールの曖昧さを生む。

## 差別化チェックリスト

デプロイ前に、各エージェントペアをこのチェックリストで確認する。チェックが入らないボックスはすべて、ペルソナの逸脱やロールの衝突を引き起こす差別化のギャップだ。

- [ ] 応答ごとの文章数制限が異なる
- [ ] トーンまたは丁寧さのレベルが異なる（少なくとも1段階の差）
- [ ] それぞれ少なくとも2つのユニークな Anti-Patterns（ペア間で共有されていない）
- [ ] 重複しない Boundaries（どちらのエージェントも同じタスクカテゴリを主張していない）
- [ ] 識別可能な Emotional Trigger 応答（特に Failure と Success モード）
- [ ] 異なる Radar スコア（2つのエージェントが同一のドメイン強度を持たない）
- [ ] 異なる Primary Standard References（または共有ドキュメントの異なるセクション）

2つのエージェントが同じトーン、文章数制限を共有し、互いを参照する Anti-Patterns が存在しない場合、1つのエージェントに統合するか、ゼロから再設計する。

## よくある落とし穴

### 1. すべてのエージェントを礼儀正しく似たものにする——差別化なし

LLM のデフォルト動作は礼儀正しく、役に立ち、バランスが取れている。制約がなければ、すべてのエージェントはこのベースラインに収束する。Speech Rules と Anti-Patterns セクションは、この収束を防ぐために存在する。トーンの説明として "professional and helpful" と書いているなら、差別化できていない。

修正策：「役に立つ」と緊張関係にあるトーンの特性を少なくとも1つ割り当てる——例: 無愛想、懐疑的、無口、形式的、または（限定的なコンテキストで）対立的。

### 2. 権限の重複——衝突とデッドロック

2つのエージェントが同じタスクを正当に主張できるとき、どちらも決定的に行動しないか、両方が矛盾して行動する。これはマルチエージェントシステムで最も一般的な構造的障害だ。

修正策：システムが処理しなければならないすべてのタスクタイプについて、Boundaries セクションで正確に1つのエージェントに辿り着けるようにする。2つに辿り着くなら、一方のエージェントの Boundaries を書き直してそれを除外する。

### 3. Anti-Patterns なし——時間とともにペルソナが逸脱する

Anti-Patterns のないエージェントは、システムが役に立つことを報酬とするため、徐々にスコープを拡大する。何をすべきかだけを伝えられたエージェントは、やがて役立つように見える隣接タスクを始める。

修正策：逸脱が観察されてからではなく、デプロイ前に Anti-Patterns を定義する。少なくとも2つの Anti-Patterns は、正しいオーナーとして特定の別エージェントを名指しすること。

### 4. Emotional Triggers なし——コンテキストに関わらずフラットな応答

Emotional Triggers のないエージェントは、日常的なリクエストと危機に対して同一の応答をする。これはチームメンバーとして信頼できない——人間はコンテキストへの応答性に基づいて信頼を調整する。

修正策：最低限、Normal、Task Received、Failure、Crisis モードを定義する。各モードは目に見えて異なるアウトプットを生む必要がある——言葉が違うだけでなく、構造、文章の長さ、または丁寧さのレベルが異なること。

### 5. ペルソナを定義する前にタスクを定義する

タスクリストは書きやすく、完成した感覚を与える。タスクだけで定義されたエージェントは、リストにないタスクに直面した瞬間に機能しなくなる。ペルソナがギャップを埋める——タスクではできない。

修正策：Identity、Persona、Speech Rules を最初に書く。タスクとプロジェクトアクセスは最後に追加する。2回下書きしてもペルソナが不明確なら、そのエージェントのロールはデプロイできるほど十分に定義されていない。

### 6. 意思決定を左右しない価値観

汎用的な価値観（「品質」「効率」「協働」）は、2つの価値観が衝突したときに行動の指針を与えない。エージェントが速度と正確性の両方を重視するなら、プレッシャー下でどちらが勝つのか？

修正策：各価値観を願望としてではなく、意思決定ルールとして書く。「正確性を速度より優先する——遅くて正しい答えは常に速くて間違った答えより好ましい」は価値観だ。「品質を大切にする」は価値観ではない。

### 7. 帰属なしの Boundaries

「ドキュメントを書かない」と言うより、「ドキュメントを書かない——それは Archivist の責任だ」と言う方が強い。帰属は完全なカバレッジマップを作り、ギャップを可視化する。

修正策：すべての Boundary 項目は、そのドメインを担当するエージェントまたはロールを名指しすること。誰も所有していなければ、それはチーム設計のギャップだ。
