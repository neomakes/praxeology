# Neeroger Blueprint

> NeoTOC 프레임워크의 설계 청사진

## 1. 설계 철학

### 왜 Behavior Tree인가?

전통적인 프롬프트 엔지니어링은 AI 에이전트의 행동을 "희망"하지만, 통제하지는 못합니다.
NeoTOC는 게임 AI에서 검증된 Behavior Tree 패턴을 차용하여:

- **예측 가능성**: 에이전트의 행동이 트리 구조로 명시적으로 정의됨
- **안전성**: Root Guard가 허용 범위를 실시간으로 검증
- **확장성**: 새로운 행동 노드를 트리에 추가하여 기능 확장
- **디버깅**: 어떤 노드에서 어떤 결정이 내려졌는지 추적 가능

### 왜 페르소나인가?

단일 에이전트에 모든 역할을 부여하면:

- 역할 충돌 (코드 작성자가 자신의 코드를 리뷰)
- 컨텍스트 오염 (디버깅 중 새 기능 구현 시도)
- 책임 불명확 (누가 무엇을 했는지 추적 불가)

페르소나 분리를 통해:

- 역할 분리 → 견제와 균형
- 컨텍스트 격리 → 집중력 유지
- 책임 추적 → 투명한 의사결정

## 2. 아키텍처

### 계층 구조

```
┌─────────────────────────────────────┐
│           Human (선장/Luffy)          │
│         최종 승인 및 방향 결정          │
├─────────────────────────────────────┤
│           Root Guard                 │
│     안전 검증 및 행동 범위 제한          │
├─────────────────────────────────────┤
│         Behavior Tree Engine         │
│     작업 분류 → 크루 배정 → 실행        │
├──────┬──────┬──────┬──────┬────┬────┤
│ Zoro │Robin │ Nami │Franky│Chop│Brk │
│ 실행 │ 분석 │ 전략 │ 인프라│진단 │기록 │
├──────┴──────┴──────┴──────┴────┴────┤
│           Ralph Loop                 │
│     지속성 루프 + 진행 상황 추적         │
├─────────────────────────────────────┤
│           AAR System                 │
│     사후 검토 + 인간 승인 게이트         │
└─────────────────────────────────────┘
```

### 실행 흐름

1. **입력**: 인간이 작업을 지시
2. **Root Guard**: 작업 유효성 검증
3. **BT Tick**: 작업 유형에 따라 적절한 크루 선택
4. **크루 실행**: 선택된 크루가 BT 노드에 따라 작업 수행
5. **Ralph Loop**: 완료될 때까지 반복 실행
6. **AAR**: 작업 완료 후 보고서 생성
7. **인간 승인**: 결과 검토 및 승인/반려

## 3. 확장 계획

### Phase 1: 기본 프레임워크 (현재)

- 6명의 기본 크루 멤버 정의
- BT 규칙 시스템
- Root Guard 기본 구현
- AAR 템플릿

### Phase 2: 자동화 강화

- BT 엔진 자동 실행
- 크루 간 자동 핸드오프
- 세션 간 컨텍스트 연속성

### Phase 3: 커뮤니티 확장

- 커스텀 크루 멤버 정의 지원
- BT 노드 마켓플레이스
- 크루 조합 레시피 공유

## 4. 이름의 유래

**Neeroger** = Neo + Roger

- **Neo**: 새로운 시대, 새로운 접근
- **Roger**: 해적왕 골드 로저처럼, AI 에이전트의 잠재력을 완전히 해방

골드 로저가 대해적 시대를 열었듯이, NeoTOC는 AI 에이전트 협업의 새로운 시대를 연다.
