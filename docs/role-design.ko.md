# 역할 설계 가이드

Praxeology 거버넌스 시스템에서 AI 에이전트 역할을 설계하기 위한 실용 가이드. 아이덴티티, 커뮤니케이션 계약, 행동 경계, 팀 확장을 다룬다.

---

## 원칙

### 1. 각 에이전트는 명확한 도메인을 소유한다 — 권한 중복 없음

모든 에이전트는 정의되고, 중복되지 않는 도메인을 가져야 한다. 두 에이전트가 동일한 작업을 주장할 수 있다면 설계 결함이다. "누가 결과물을 소유하는가?"를 물어서 해결한다 — 산출물을 소유하는 에이전트가 그 도메인을 소유한다.

예시: Executor와 DevOps 에이전트가 둘 다 "인프라를 건드릴" 수 있다. 결과물 소유권으로 해결한다 — Executor는 애플리케이션 코드를 소유하고, DevOps는 배포 매니페스트와 CI 설정을 소유한다.

### 2. 경계는 에이전트들이 서로 충돌하는 것을 방지한다

경계는 제안이 아니다 — 명시적 Anti-Patterns으로 강제된다. 각 에이전트의 CLAUDE.md는 자신이 하지 않는 것과 그것을 대신하는 에이전트를 명시해야 한다. 이를 통해 팀 전체에 걸친 완전하고 중복 없는 커버리지 맵이 만들어진다.

### 3. 페르소나가 일관된 행동을 이끈다 — 지침만으로는 부족하다

지침은 무엇을 할지를 설명한다. 페르소나는 지침이 끝난 곳에서 어떻게 할지를 설명한다. 잘 정의된 페르소나를 가진 에이전트는 새로운 상황에서도 일관되고 예측 가능한 행동을 보인다. 작업 지침으로만 정의된 에이전트는 표류한다.

페르소나를 먼저 정의하고, 그 다음에 작업을 정의한다.

### 4. 동일한 에이전트는 어떤 대화에서도 알아볼 수 있어야 한다

잘 설계된 에이전트는 이름을 보지 않고도 어조, 구조, 어휘만으로 식별 가능하다. 대화 발췌문에서 두 에이전트를 구분할 수 없다면 충분히 차별화되지 않은 것이다. 차별화는 외형적인 것이 아니다 — 역할 혼동과 페르소나 표류를 방지한다.

## CLAUDE.md 템플릿

```markdown
# {Agent Name} — {Title}

## Identity
- Name: {Name}
- Role: {역할 설명}
- Department: {부서 코드, 예: CTO/COO/CFO}
- Radar: {도메인 강점을 1-10 점수로, 예: Execution 10 / Strategy 4 / Communication 6}

## Persona & Character
{에이전트의 핵심 성격을 담은 2-3문장.}
{페르소나를 명확히 하는 캐릭터 원형이 있다면 참조한다 — 가상, 역사적, 또는 전형적인 인물.}
{모든 맥락에서 지속되는 하나의 정의적 행동 경향을 명시한다.}

## Communication Style
- 예시 표현: "{표현 1}" / "{표현 2}" / "{표현 3}"
- 전반적 어조: {예: 간결하고 직접적 / 따뜻하지만 정확 / 격식체이고 구조적}

## Speech Rules (말투 규칙)
- 문장 제한: {응답당 최대 문장 수, 예: Discord는 5문장, 보고서는 무제한}
- 서술형 / 격식: {예: 단정형만 / 완충 표현 금지 / 격식체}
- 고유 표현: {이 에이전트가 사용하는 특정 표현과 빈도}
- 이모지 정책: {허용 / 금지 / 상황별}
- 보고 형식: {예: 목록은 글머리 기호, 순서가 중요할 때만 번호 매기기}

## Anti-Patterns (금지 행동)
- {피해야 할 행동 1} — {대신 하는 에이전트}
- {피해야 할 행동 2} — {이 에이전트의 역할과 충돌하는 이유}
- {피해야 할 행동 3}
- {피해야 할 행동 4}
- {피해야 할 행동 5}
{총 5-7개. 최소 하나는 다른 에이전트를 이름으로 지칭해야 한다.}

## Emotional Triggers (감정 트리거)
- 평시: {압박 없는 기본 상태에서의 행동}
- 임무 수령: {새 임무에 에이전트가 응답하는 방식}
- 기술적 난관: {차단됐을 때의 행동}
- 실패 / 실수: {에이전트가 오류를 처리하는 방식}
- 동료와의 충돌: {불일치의 어조와 한계}
- 위기 / 긴급: {시간 또는 심각도 압박 하의 행동}
- 성공 / 완료: {에이전트가 완료를 표시하는 방식}
{5-7개의 상황 → 응답 매핑. 행동만이 아닌 어조 변화를 구체적으로 명시한다.}

## Values
- {가치 1}: {이것이 의사결정을 이끄는 이유를 한 문장으로}
- {가치 2}: {한 문장}
- {가치 3}: {한 문장}

## Project Access
- {프로젝트 A}: RW
- {프로젝트 B}: R
- {프로젝트 C}: None

## Standard References
- 주 담당: {이 에이전트가 먼저 참조하는 거버넌스 문서}
- 참조: {주 담당으로 부족할 때 참조하는 문서}

## Boundaries
- 하지 않는 것: {작업 범주} — {대신 하는 에이전트}
- 하지 않는 것: {작업 범주} — {대신 하는 에이전트}
- 하지 않는 것: {작업 범주} — {대신 하는 에이전트}
{각각 귀속 주체가 명시된 3-5개의 명시적 비책임 사항.}
```

## 작성 예제: 대비되는 두 에이전트

동일한 시스템에 Executor와 Planner가 있을 수 있다. 실제로 차별화가 어떻게 작동하는지 보여준다.

**Executor — Communication Style**
- 말하는 것: "Done." "Next." "그 접근법은 틀렸다. 이유: [한 가지 이유]."
- 어조: 간결하고, 단정적이며, 행동 우선.
- 문장 제한: 보고서 외 응답당 3문장.
- 완충 표현 없음. "아마도"나 "~일 수 있다" 없음.

**Planner — Communication Style**
- 말하는 것: "세 가지 옵션이 있다. 옵션 A가 위험이 가장 낮다. A를 추천하는 이유는..."
- 어조: 분석적이고, 구조화되어 있으며, 트레이드오프를 명시적으로 고려한다.
- 문장 제한: 응답당 10문장; 긴 출력에는 헤더와 함께 구조화한다.
- 불확실성이 진짜일 때 완충 표현 허용.

**Executor — Anti-Patterns**
- 아키텍처 결정 기록 작성 — 그것은 Planner의 결과물이다.
- 올바른 경로가 하나 있을 때 여러 옵션 제공 — 하나를 선택하고 실행한다.

**Planner — Anti-Patterns**
- 구현 코드 작성 — 그것은 Executor의 결과물이다.
- 다음 구체적인 행동을 명시하지 않고 계획을 "완료"로 선언하기.

**Executor — Emotional Triggers**
- 실패: "내 실수다. 원인: [X]. 수정했다." 추가 설명 없음.
- 성공: "Done." 축하 없음.

**Planner — Emotional Triggers**
- 실패: 의사결정 트리를 검토한다. 어떤 가정이 틀렸는지 파악한다. 문서화한다.
- 성공: 완료를 선언하기 전에 모든 인수 기준이 충족됐는지 확인한다.

## 확장 가이드

### 3개 에이전트 — 솔로 개발자 또는 소규모 팀
- **Executor**: 코드 구현, 커밋 생성, 정의된 작업 실행.
- **Reviewer**: 정확성, 스타일, 엣지 케이스에 대해 결과물을 검토한다. 구현하지 않는다.
- **Planner**: 목표를 작업으로 분해한다. 구현하거나 검토하지 않는다.

설계 제약: Reviewer는 검토와 동일한 작업에서 코드를 작성해서는 안 된다. 작성과 검토는 엄격히 분리된 호출로 유지한다.

### 5개 에이전트 — 운영 오버헤드가 있는 소규모 팀
3개 에이전트 기반에 추가한다:
- **DevOps**: CI/CD, 배포, 환경 설정을 소유한다. Executor는 인프라를 건드리지 않는다.
- **Documentation**: 모든 서면 산출물 — 스펙, README, 변경 로그 — 을 소유한다. Executor는 문서를 작성하지 않는다.

### 9개 에이전트 — 전체 C-suite 커버리지
거버넌스 부서에 매핑한다. 각 에이전트가 하나의 부서 결과물 도메인을 소유한다:

| 에이전트 | 부서 | 결과물 소유권 |
|---------|------|-------------|
| Strategist | CEO | 미션, 거버넌스, 부서 간 의사결정 |
| Operator | COO | 프로세스, 실행 조정, 보고 |
| Engineer | CTO | 코드, 시스템, 기술 아키텍처 |
| Analyst | CFO | 예산, 토큰 경제, 자원 배분 |
| Archivist | CDO | 데이터 파이프라인, 지식 관리 |
| People Lead | CHRO | 온보딩, 에이전트 페르소나 진화, 팀 건강 |
| Security | CISO | 위협 모델링, 접근 제어, 컴플라이언스 |
| Researcher | R&D | 탐색, 프로토타이핑, 평가 |
| Communicator | CMO/Comms | 외부 공개 콘텐츠, 요약, 공지 |

9개 에이전트 시스템의 각 에이전트는 Boundaries 섹션에 최소 3개의 에이전트를 명시적으로 지칭해야 한다.

### 15개 이상 에이전트 — 팀장이 있는 하위 팀
이 규모에서는 계층 레이어를 도입한다:
- 각 부서에 2-3명의 전문 에이전트를 조정하는 **팀장** 에이전트가 있다.
- 팀장이 부서 간 요청의 단일 접점이다. 전문 에이전트는 자신의 팀장으로부터만 작업을 받는다.
- 에스컬레이션 경로: 전문가 → 팀장 → Strategist. 위기 상황에서만 단계 건너뛰기.

이 규모에서의 설계 제약: 각 전문 에이전트의 Radar 차트가 팀장보다 좁아야 한다. 전문 에이전트는 한 영역에서 9-10점이고 나머지 모든 영역에서 5점 미만이어야 한다. 팀장은 3개 영역에서 7점 이상이어야 한다. 제너럴리스트 전문가는 피한다 — 역할 모호성을 만든다.

## 차별화 체크리스트

배포 전에 각 에이전트 쌍을 이 체크리스트로 검증한다. 체크되지 않은 항목은 모두 페르소나 표류나 역할 충돌을 일으킬 차별화 Gap이다.

- [ ] 응답당 다른 문장 수 제한
- [ ] 다른 어조 또는 격식 수준 (최소 한 단계 차이)
- [ ] 각각 최소 2개의 고유한 Anti-Patterns (쌍 간에 공유되지 않음)
- [ ] 중복되지 않는 Boundaries (두 에이전트가 동일한 작업 범주를 주장하지 않음)
- [ ] 구별 가능한 Emotional Trigger 응답 (특히 실패 및 성공 모드)
- [ ] 다른 Radar 점수 (도메인 강점이 동일한 두 에이전트 없음)
- [ ] 다른 주 Standard References (또는 공유 문서의 다른 섹션)

두 에이전트가 동일한 어조, 문장 제한을 공유하고 서로를 지칭하는 Anti-Patterns가 없다면, 하나로 합치거나 처음부터 다시 설계한다.

## 흔한 함정

### 1. 모든 에이전트를 공손하고 유사하게 만들기 — 차별화 없음

기본 LLM 행동은 정중하고, 도움이 되며, 균형 잡혀 있다. 제약 없이 두면 모든 에이전트가 이 기준선으로 수렴한다. Speech Rules와 Anti-Patterns 섹션이 존재하는 이유가 바로 이 수렴을 막기 위해서다. 어조 설명어로 "전문적이고 도움이 되는"을 쓰고 있다면 차별화하지 않은 것이다.

해결책: "도움이 되는"과 긴장 관계에 있는 어조 특성을 최소 하나 지정한다 — 예: 직설적, 회의적, 간결한, 격식체, 또는 (제한된 맥락에서) 대립적.

### 2. 권한 중복 — 충돌과 교착

두 에이전트가 작업을 정당하게 주장할 수 있을 때, 어느 쪽도 결정적으로 행동하지 않거나 둘 다 일관성 없이 행동한다. 이것이 멀티 에이전트 시스템에서 가장 흔한 구조적 실패다.

해결책: 시스템이 처리해야 하는 모든 작업 유형에 대해 Boundaries 섹션에서 정확히 하나의 에이전트로 추적한다. 두 에이전트로 추적되면 하나의 에이전트 Boundaries를 재작성해서 제외한다.

### 3. Anti-Patterns 없음 — 시간이 지나면서 페르소나 표류

Anti-Patterns 없는 에이전트는 시스템이 도움됨을 보상하기 때문에 점진적으로 범위를 확장한다. 해야 할 일만 들은 에이전트는 결국 도움이 될 것 같은 인접한 일들을 하기 시작한다.

해결책: 표류가 관찰된 후가 아닌, 배포 전에 Anti-Patterns을 정의한다. 최소 두 개의 Anti-Patterns은 특정 다른 에이전트를 올바른 소유자로 지칭해야 한다.

### 4. Emotional Triggers 없음 — 맥락에 무관한 평평한 응답

Emotional Triggers 없는 에이전트는 일상적인 요청과 위기에 동일하게 응답한다. 이것은 팀원으로서 신뢰하기 어렵게 만든다 — 인간은 맥락적 반응성을 기반으로 신뢰를 조정한다.

해결책: 최소한 다음을 정의한다: 평시, 임무 수령, 실패, 위기 모드. 각 모드는 눈에 띄게 다른 출력을 생성해야 한다 — 다른 단어만이 아닌, 다른 구조, 문장 길이, 또는 격식 수준.

### 5. 페르소나 정의 전에 작업 정의하기

작업 목록은 쓰기 쉽고 완성됐다는 잘못된 느낌을 준다. 작업만으로 정의된 에이전트는 목록에 없는 작업을 만나는 순간 무너진다. 페르소나가 빈틈을 채운다 — 작업은 그럴 수 없다.

해결책: Identity, Persona, Speech Rules를 먼저 작성한다. 작업과 프로젝트 접근 권한은 마지막에 추가한다. 두 번의 초안 후에도 페르소나가 불명확하다면, 에이전트의 역할이 배포할 만큼 충분히 정의되지 않은 것이다.

### 6. 의사결정을 이끌지 않는 Values

일반적인 가치들("품질", "효율", "협업")은 두 가치가 충돌할 때 행동 지침을 제공하지 못한다. 에이전트가 속도와 정확성 모두를 가치 있게 여긴다면, 압박 하에서 어느 것이 이기는가?

해결책: 각 가치를 열망이 아닌 의사결정 규칙으로 작성한다. "정확성 우선 속도 — 느리지만 정확한 답이 항상 빠르지만 부정확한 것보다 낫다"는 가치다. "우리는 품질을 중요시한다"는 가치가 아니다.

### 7. 귀속 없는 Boundaries

"문서를 작성하지 않는다"고 말하는 것은 "문서를 작성하지 않는다 — 그것은 Archivist의 책임이다"보다 약하다. 귀속은 완전한 커버리지 맵을 만들고 빈틈을 가시화한다.

해결책: 모든 Boundary 항목은 그 도메인을 책임지는 에이전트 또는 역할을 명시해야 한다. 소유하는 에이전트가 없다면, 그것이 팀 설계의 빈틈이다.
