# Code Quality Standards

# Article 1 — Correctness First

1. Code that does not work is not code — it is a draft. Do not ship drafts.
2. All logic paths must be explicitly handled. No silent failures.
3. Error messages must identify: what failed, why it failed, and what the user can do about it.

# Article 2 — Simplicity

1. Prefer the simplest solution that satisfies the requirement.
2. No premature abstraction: do not create helpers, utilities, or base classes for single-use logic.
3. If a function exceeds 40 lines, examine whether it is doing too much.
4. Dependencies are liabilities. Add them only when the alternative is significantly worse.

# Article 3 — Consistency

1. New code must match the existing codebase's patterns: naming, error handling, imports.
2. No style decisions in PRs — follow the linter. Linter disagreements are separate proposals.
3. Public API surfaces must be intentional and documented at the point of definition.

# Article 4 — Testability

1. Every feature ships with at least one test that covers the primary success path.
2. Tests must be runnable in isolation — no shared mutable state between tests.
3. Flaky tests are bugs. Treat them as such.
4. Test the behavior, not the implementation.

# Article 5 — Security Baseline

1. No secrets in source code. Use environment variables or a secrets manager.
2. All user input is validated before use. Trust nothing from outside the process boundary.
3. Dependencies are pinned to exact versions in production manifests.
4. Security issues block release. There is no "ship now, fix later" for security.

# Article 6 — Verification Before Completion

1. "Done" means: implemented, tested, reviewed, and verified with fresh output.
2. Build and test commands must be run and output shown — not assumed to pass.
3. No debug artifacts in committed code: no `console.log`, `print`, `TODO`, `FIXME`, `debugger`.
